## 数据模型

### 三层表结构

```
                    知识图谱 (AGE 图)
                    ┌───────────────────────┐
                    │  Vertex (Object)     │
                    │  ┌─────────────────┐ │     ┌─────────────────┐
                    │  │ name: "张三"     │ │────▶│ Metric (销售额)  │
                    │  │ type: "employee" │ │     └────────┬────────┘
                    │  │ employee_id:..  │ │              │
                    │  └─────────────────┘ │              ▼
                    └───────────────────────┘     ┌─────────────────┐
                                                  │ Dimension (上月) │
                                                  └─────────────────┘
                           │
            ┌──────────────┼──────────────┐
            │  (graph_name, vertex_id)    │
            ▼                             ▼
    向量侧表 (ontosql)              关联物化表 (ontosql)
    ┌─────────────────────┐          ┌──────────────────────────┐
    │ vertex_embeddings   │          │ object_attribute_mapping │
    │ ├ vertex_id         │          │ ├ graph_name             │
    │ ├ vertex_name       │◀─────────│ ├ object_vertex_id       │
    │ ├ label_name        │  关联    │ ├ object_label           │
    │ └ embedding         │          │ ├ attr_id ───────────────│──▶ attribute_embeddings
    └─────────────────────┘          │ ├ relation_type          │    ├ attr_name

```

### 关键设计决策

| 决策 | 方案 | 理由 |
|------|------|------|
| 向量与图分开存储 | 独立侧表 | AGE 的 agtype 将属性序列化为单列，无法建向量索引；分离后 pgvector 可对独立列建 HNSW 索引 |
| 属性是否建模为图顶点 | 看场景 | Metric/Dimension 类属性建议建顶点（支持图验证），Entity 字段（如 age, phone）不需要 |
| 关联物化表 | object_attribute_mapping | 避免每次 read Cypher MATCH 的开销（跨多次 JOIN），提升 is_verified 验证性能 |
| 多路召回权重 | 向量 0.6 + trigram 0.4 | 向量语义泛化能力强，trigram 补充精确字符匹配；可调参数 |
| AGE 版本 | master（PG 18 API）→ 版本守卫 | master 功能最新但面向 PG 18，通过 `#if PG_VERSION_NUM` 条件编译兼容 PG 17 |

## 查询流程

### 从 NL 到 SQL 的完整链路

```
用户输入: "张三上月销售额多少"
         │
    ┌────▼────┐
    │ NLP 层   │  大模型 (GPT-4 / DeepSeek / Qwen)
    │ 实体提取 │  ──▶ 提取: 对象=["张三"], 属性=["销售额"], 维度=["上月"]
    │ 意图识别 │
    └────┬────┘
         │
    ┌────▼────────────────────────────────┐
    │ OntoSQL 检索层 (SQL 函数)           │
    │                                     │
    │ search_object_attribute("张三上月..")│
    │   │                                 │
    │   ├─ search_objects("张三上月..")   │
    │   │     → [张三(0.95), 钱七(0.35)]  │
    │   │                                 │
    │   ├─ search_attributes("张三上月..") │
    │   │     → [销售额(0.97), 客单价(0.57)]│
    │   │                                 │
    │   └─ CROSS JOIN + 关联验证          │
    │         → (张三, 销售额, is_verified=true, score=0.96) │
    │         → (张三, 客单价, is_verified=false, score=0.76) │
    └────┬────────────────────────────────┘
         │  (仅保留 is_verified=true)
    ┌────▼────────────────────────────────┐
    │ 图查询层 (AGE Cypher)               │
    │                                     │
    │ MATCH (张三:{Object})              │
    │   -[:HAS_METRIC]->(销售额:{Metric}) │
    │   -[:HAS_DIMENSION]->(上月:{Dim})   │
    │                                     │
    │ → 拿到实际数据表/字段 → 生成 SQL    │
    └────┬────────────────────────────────┘
         │
    ┌────▼────┐
    │ 执行 SQL │  SELECT SUM(amount)
    │ 返回结果 │  FROM sales_fact
    └─────────┘  WHERE employee_id='E001'
                   AND period='2025-04'
```

### 多路召回算法详解

```
输入: query_text = "销首额" (含错字), p_query_embedding = vec("销首额")

  向量召回路径 (HNSW ANN)               trigram 召回路径 (GIN + pg_trgm)
  ┌────────────────────────┐           ┌────────────────────────────┐
  │ <=> 余弦距离 < 阈值    │           │ similarity > pg_trgm.     │
  │ 语义匹配: 销首额 ≈ 销售额│           │   similarity_threshold      │
  │ (向量不受错字影响)     │           │ 字符级: "销售额" % "销首额" │
  │                        │           │ → 相似度 ~0.2              │
  │ 结果:                   │           │                            │
  │ 销售额  0.85            │           │ 结果:                       │
  │ 客单价  0.62            │           │ 销售额  0.20               │
  └───────────┬────────────┘           └──────────────┬─────────────┘
              │                                       │
              └──────────┬────────────────────────────┘
                         ▼
              ┌─────────────────────┐
              │   UNION ALL 合并    │
              │   GROUP BY 去重     │
              │   加权综合分:        │
              │   0.6×向量 + 0.4×trigram│
              └──────────┬──────────┘
                         ▼
              排名后的候选列表 (Top-K)
              销售额  (0.85×0.6 + 0.20×0.4 = 0.59)
              客单价  (0.62×0.6 + 0.00×0.4 = 0.37)
```

## 扩展架构 (PGXS)

OntoSQL 通过 PostgreSQL 的 PGXS（Extension Build Infrastructure）机制编译和安装扩展：

```
pgvector (PGXS 标准扩展)
  ├─ Makefile  ──▶ PG_CONFIG=.../pg_config  ──▶ 编译 → 安装到 $PG_HOME/{lib,share}/
  └─ vector.control + vector--*.sql ──▶ CREATE EXTENSION vector

Apache AGE (PGXS 标准扩展)
  ├─ Makefile  ──▶ PG_CONFIG=.../pg_config  ──▶ 编译 → 安装到 $PG_HOME/{lib,share}/
  └─ age.control + age--*.sql ──▶ CREATE EXTENSION age
```

两个扩展均为标准 PGXS 扩展，互不依赖，可独立安装升级。OntoSQL Schema 依赖两者均已加载。

## 安全架构

```
客户端
  │
  ├─── 本地连接 (Unix Socket)
  │     认证: trust (免密)
  │
  ├─── 远程连接 (TCP)
  │     认证: md5 密码
  │     加密: 可选 SSL (编译已启用 OpenSSL)
  │
  └─── Docker 连接
        端口: 5432 (通过环境变量可改)
        安全: no-new-privileges + 资源限制
```

生产环境建议：
- 将 `pg_hba.conf` 的 `host ... trust` 改为 `md5` 或 `scram-sha-256`
- 启用 SSL 连接 (`ssl = on`)
- 使用专用非 root 用户运行 PG 进程
- 定期更新依赖组件和扩展
