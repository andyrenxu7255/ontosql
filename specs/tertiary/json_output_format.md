# JSON Output Structure Format
> Tier 3 Retrieval Document | ≤2000 tokens | id: json_output_format
>
> This document governs ALL JSON output generated by OntoSQL Skills.
> Every Skill's stdout must conform to exactly one of the envelopes below.

## Envelope A: Success Response

Applied when a Skill completes without error. All Skills use this exact structure.

```json
{
  "status": "success",
  "skill": "<skill-name>",
  "timestamp": "<ISO 8601 UTC>",
  "data": { },
  "meta": {
    "elapsed_ms": 0
  }
}
```

### Field Rules (non-negotiable)

| Field | Type | Required | Constraint |
|-------|------|----------|------------|
| `status` | string | YES | Literal `"success"` (lowercase) |
| `skill` | string | YES | Exact skill name as in manifest.json |
| `timestamp` | string | YES | ISO 8601 with Z suffix: `2026-05-11T10:00:00Z` |
| `data` | object\|array | YES | Empty `{}` or `[]` if no results, never `null` or omitted |
| `meta.elapsed_ms` | integer | YES | Milliseconds since Skill start, never negative |

### Outcome-scoped `data` examples

**Query Skills** (`data` is always an object with `results` array + `count`):
```json
"data": {
  "results": [
    {"vertex_id": 1, "vertex_name": "张三", "combined_score": 0.9234}
  ],
  "count": 1
}
```

**Lifecycle Skills** (`data` is always an object with contextual fields):
```json
"data": {"running": true, "port": 5432, "uptime_seconds": 3600}
```

**Write Skills** (`data` confirms action):
```json
"data": {"vertex_id": 42, "vertex_name": "RegionA", "graph_name": "ontosql_graph"}
```

## Envelope B: Error Response

Applied when any validation or execution failure occurs.

```json
{
  "status": "error",
  "skill": "<skill-name>",
  "timestamp": "<ISO 8601 UTC>",
  "error": {
    "code": "ERR_XXX",
    "message": "<human-readable string>",
    "details": { }
  },
  "meta": {
    "elapsed_ms": 0
  }
}
```

### Error Field Rules

| Field | Type | Required | Constraint |
|-------|------|----------|------------|
| `error.code` | string | YES | Format `ERR_<CATEGORY>_<SPECIFIC>`, uppercase, underscore-delimited |
| `error.message` | string | YES | No full stops at end; no SQL internals; max 200 chars |
| `error.details` | object | YES | Key-value diagnostic data, empty `{}` if no details, never `null` |

### Exit Code → Error Code Mapping

| Exit Code | error.code Pattern | Meaning |
|-----------|-------------------|---------|
| 1 | `ERR_INPUT_*` | Client-side input validation failure |
| 2 | `ERR_PG_*` `ERR_QUERY_*` `ERR_WRITE_*` `ERR_CYPHER_*` | Server-side execution failure |
| 3 | `ERR_CONFIG_*` | Environment/config missing |
| 4 | `ERR_SKILL_NOT_FOUND` | Skill name unknown |

### Key Constraints

1. NEVER emit bare text to stdout. Only the JSON envelope.
2. NEVER use `null` for `data` or `error.details`. Empty objects required.
3. NEVER nest error inside data. Error and data are mutually exclusive.
4. ALWAYS include `elapsed_ms` even when it is 0.
5. ALWAYS quote all string values. No bare literals.
6. ALWAYS use double quotes (`"`), NEVER single quotes (`'`) for JSON keys/values.
7. ALWAYS terminate complete object. No truncation mid-key or mid-value.
